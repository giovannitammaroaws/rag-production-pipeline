# ADR 003 - Session Deletion Strategy on Right to Be Forgotten

**Status:** Accepted
**Date:** 2026-03-19

---

## Context

GDPR art. 17 gives users the right to request deletion of their data. In this system that means: a user uploads a PDF, later asks for it to be deleted, and all traces of that document must be erased.

### What the right to be forgotten covers (mandatory, immediate)

When `DELETE /documents/{doc_id}` is called, these four deletions are not optional:

```
S3          → the original PDF file
Aurora      → all chunks and vectors generated from that PDF
              DELETE FROM chunks WHERE metadata->>'doc_id' = $1
DynamoDB    → the document entry in the documents table
DynamoDB    → the ingestion job history in the jobs table
```

This is the right to be forgotten. It applies to the original document and all its processed derivatives (chunks, vectors). These are deleted immediately and synchronously within the same Lambda invocation.

---

## The Side Effect: Session History

After the document is deleted, a problem remains. Conversation history in DynamoDB may contain assistant answers that were generated using content from the deleted document:

```
session turn (still in DynamoDB after document deletion):

  user:      "What is the 2025 revenue?"
  assistant: "The 2025 revenue was $4.2B..."
             ↑
             this answer was generated from chunks of the deleted document
```

This assistant response is not the original document - it is text generated by Claude. Whether it falls under GDPR art. 17 is legally debated. However, a conservative interpretation says it should be removed too, since it is a derivative of the deleted data.

The challenge is technical: sessions are stored with `session_id` as partition key and `turn_id` as sort key. There is no `doc_id` field in session turns - no direct link between an answer and the source document that contributed to it.

---

## Options for Session Deletion

### Option A - TTL expiry (chosen)

Session turns already have a `ttl` attribute set to 24 hours from creation. DynamoDB deletes expired items automatically. Do nothing extra at delete time.

**Pro:**
- Zero additional code in the delete flow
- Zero cost

**Con:**
- Up to 24 hours where session history still contains answers from deleted content
- DynamoDB TTL deletion is not guaranteed to the second

**Acceptable because:** new queries will never retrieve content from the deleted document - it is gone from S3 and Aurora immediately. Only pre-existing chat turns may reference it, and only for up to 24 hours.

---

### Option B - Store source doc_ids in each session turn

At retrieval time, record which documents contributed to each answer:

```json
{
  "session_id": "sess-abc",
  "turn_id": "2026-03-19T10:05:00.000Z",
  "role": "assistant",
  "content": "The 2025 revenue was $4.2B...",
  "source_doc_ids": ["uuid-1234"],
  "ttl": 1742486700
}
```

At delete time, query a mapping table to find and delete affected turns immediately.

**Pro:** immediate deletion, fully auditable

**Con:** schema change, extra complexity in the retrieval Lambda, DynamoDB does not support array-valued GSIs natively - requires a separate `doc_id → session_id` mapping table

**When this becomes necessary:** if session TTL is extended beyond 24 hours, or if compliance auditors require proof of immediate erasure of derived content.

---

### Option C - Full DynamoDB scan

Scan the entire sessions table at delete time and remove matching turns.

**Pro:** immediate, no schema changes

**Con:** full table scan is expensive and grows linearly with table size. Rejected.

---

## Decision

**Option A - TTL expiry.**

The document and all its direct derivatives (S3, Aurora, DynamoDB metadata) are deleted immediately. Session history - which contains derived text, not the original document - expires within 24 hours via TTL.

This decision must be disclosed in the privacy policy: *"conversation history referencing deleted documents may persist for up to 24 hours before automatic expiry."*

If requirements change, migrate to Option B by adding `source_doc_ids` to the session schema.
